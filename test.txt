### Challenges Faced

Solution:
- Profiled the queries
- Realized we were doing brute-force similarity search
- Switched to FAISS with HNSW index
- Added caching layer for frequently accessed embeddings
- Result: 5-10 seconds to 200-500ms"

**Challenge 2: Agent Hallucination**

"Early on, agents would confidently state facts that weren't in the documents.

Solution:
- Implemented strict retrieval-based generation
- Every fact had to cite a source document
- Added validation agent to cross-check facts
- Implemented confidence thresholds
- Result: Hallucination rate dropped from 8% to <0.5%"

**Challenge 3: Multi-Agent Coordination**

"Getting three agents to work together was tricky. Initially, agents would redo work and get stuck in loops.

Solution:
- Implemented supervisor agent pattern
- Agents communicate through structured messages
- Added step limits to prevent infinite loops
- Result: Clean workflows, no redundant work"

**Challenge 4: State Management**

"At scale, managing agent state became complex.

Solution:
- Designed a comprehensive checkpoint structure
- Used JSON serialization for agent state
- Implemented transaction-like semantics
- Used PostgreSQL JSONB for flexible storage
- Result: Reliable state management across failures"

# SECTION 7: RENTALHUNT PROJECT - DEEP DIVE

## Project Overview

"RentalHunt is a real estate SaaS platform serving Canada, UAE, and Europe.

Users:
1. Landlords - List/manage properties, track rent, handle maintenance
2. Tenants - Search/apply for properties, make payments, report issues
3. Admins - Monitor platform, manage disputes, revenue tracking

Regional Requirements:
- Canada: CAD, English/French, strict tenant protections
- UAE: AED, English/Arabic, landlord-favorable regulations
- Europe: EUR, multiple languages, GDPR compliance"

## Technology Stack

Layer            | Technology                     | Purpose
-----------------|--------------------------------|----------------------------------
Backend          | FastAPI 0.104+                 | Async Python framework, type-safe
ORM              | SQLAlchemy 2.0                 | Database abstraction, migrations
API Models       | Pydantic                       | Request/response validation
Database         | PostgreSQL 14+                 | ACID compliance, complex queries
Caching          | Redis 7.0+                     | Session cache, property cache
Authentication   | Firebase Auth                  | User management, JWT tokens
Async Tasks      | Celery + Redis                 | Email notifications, bulk operations
Containerization | Docker                         | Consistent environments
Cloud            | AWS (EC2, RDS, S3, CloudFront) | Hosting, storage, CDN

## Dynamic Database Routing Middleware

"The core architectural innovation. Each region has its own database.
Solution:
1. Extract region from JWT token
2. Extract region from subdomain
3. Validate region matches
4. Create region-specific database session
5. Store in FastAPI request context (request.state.db)
6. All subsequent code uses request.state.db

Benefits:
- Automatic routing (no manual parameter passing)
- Type safe (SQLAlchemy session bound to region)
- Enforced at middleware level (can't accidentally leak data)
- Easy to test (inject test database)"

## Caching Strategy

"Multi-level caching:

Level 1: HTTP Cache Headers
- Properties: max-age=3600 (1 hour)
- User profiles: max-age=300 (5 minutes)
- Search results: max-age=60 (1 minute)

Level 2: Redis Cache (Cache-Aside pattern)
- Try Redis first
- Cache miss: hit database, store in Redis
- Region-aware cache keys (include region in key!)

Level 3: Application-Level Caching
- Cache expensive search computations
- 5-minute TTL for search results

Performance Impact:
- Property browsing: 50-100ms to 15-20ms (60-70% improvement)
- Search: 200-500ms to 30-100ms (60-70% improvement)
- Overall P99 latency: Reduced from 800ms to 300ms"

## Authentication & Authorization

"Firebase Integration:
1. User enters credentials on frontend
2. Frontend calls Firebase Authentication
3. Firebase returns ID token + Refresh token
4. Backend validates token
5. Backend creates session in Redis
6. User accesses protected endpoints

Role-Based Access Control (RBAC):
- LANDLORD: Can list/manage properties
- TENANT: Can search/apply for properties
- ADMIN: Can manage users, moderate disputes"

## Critical Issues Solved

### Issue 1: Database Connection Pool Exhaustion

"Problem: System would crash with 'no connections available' every few hours.

Root Cause: Background tasks created database sessions but weren't returning connections.

Solution:
- Use dependency injection for database sessions
- Ensure sessions closed with context managers
- Monitor connection usage in production

Result: Stable for weeks, connection pool never exceeds 12"

### Issue 2: N+1 Query Problem

"Problem: Endpoint listing 100 properties took 10+ seconds.
Was doing 1 query for properties + 100 queries for each landlord = 101 queries.

Solution: SQLAlchemy eager loading with joinedload()
Result: 10+ seconds to 200ms (50x improvement)"

### Issue 3: Cache Invalidation Bug

"Problem: When landlord updated property, tenants still saw old cached data.

Root Cause: Race condition + multiple Redis instances per region.

Solution:
- Redis Pub/Sub for cache invalidation across instances
- Atomic pipeline operations
- Cache warming after updates

Result: Real-time updates, <100ms to show new data"

### Issue 4: Search Performance Degradation

"Problem: Complex searches took 5-10 seconds. Full table scan.

Solution:
CREATE INDEX idx_property_search ON properties(region, status, bedrooms, price, created_at DESC)

Result: 5-10 seconds to 200-300ms (30x improvement)"

"My principles:

"Horizontal scaling:

- Better indexes
- Query optimization

- Single server: 1000 req/sec

## PYTHON & BACKEND QUESTIONS

**Q1: What's the difference between list and tuple in Python?**

"Lists are mutable, tuples are immutable.

Practical difference:
- List: Can change items after creation
- Tuple: Cannot change items

Why it matters:
- Tuples can be dictionary keys (lists can't)
- Performance: Tuples slightly faster
- Used in function returns for multiple values

Example:
my_list = [1, 2, 3]
my_list[0] = 10  # OK

my_tuple = (1, 2, 3)
my_tuple[0] = 10  # TypeError!"

---

**Q2: What is a decorator in Python?**

"A decorator is a function that modifies another function or class.

Real-world example:

def require_auth(f):
    def wrapper(*args, **kwargs):
        token = request.headers.get('Authorization')
        if not token:
            raise HTTPException(401)
        return f(*args, **kwargs)
    return wrapper

@require_auth
@app.get('/protected')
async def protected_endpoint():
    return {'message': 'only for authenticated users'}

Used extensively in FastAPI, Django, etc."

---

**Q3: What's the difference between async and sync functions?**

"Sync functions block. Async functions don't.

In RentalHunt, switching to FastAPI (async) gave us ability to handle 10x more concurrent users.

Sync:
- 1000 requests * 200ms each = 200 seconds total

Async:
- 1000 requests all start simultaneously
- All 1000 finish in ~200ms"

---

**Q4: What are Python generators?**

"Generators produce values lazily (on demand) instead of all at once.

Example:
def read_large_file(file_path):
    with open(file_path, 'r') as f:
        for line in f:
            yield line  # One line at a time

for line in read_large_file('huge_file.txt'):
    process(line)  # Memory efficient!

Used in document processing pipelines."

---

**Q5: What is the GIL in Python?**

"Global Interpreter Lock. Only one thread executes Python code at a time.

Impact:
- CPU-bound tasks: Threading doesn't help (use multiprocessing)
- IO-bound tasks: Threading works (thread waits for IO, others run)
- Best solution: AsyncIO for IO-bound workloads

In my projects, I use AsyncIO for database queries and API calls."

---

**Q6: What's the difference between deepcopy and copy?**

"copy: Shallow copy (nested objects still shared)
deepcopy: Deep copy (completely independent)

Example:
import copy

original = {'a': [1, 2, 3]}
shallow = copy.copy(original)
deep = copy.deepcopy(original)

original['a'].append(4)
print(shallow['a'])  # [1, 2, 3, 4] - affected!
print(deep['a'])     # [1, 2, 3] - not affected"

---

**Q7: What is *args and **kwargs?**

"*args: Variable positional arguments (tuple)
**kwargs: Variable keyword arguments (dict)

Example:
def flexible_function(*args, **kwargs):
    for arg in args:
        print(arg)
    for key, value in kwargs.items():
        print(f'{key}: {value}')

flexible_function(1, 2, 3, name='Avinash', city='Pune')
Output:
1, 2, 3
name: Avinash
city: Pune"

---

**Q8: What are context managers?**

"Context managers handle setup and cleanup automatically.

Example:
with open('file.txt', 'r') as f:
    content = f.read()
# File automatically closed, even if exception occurs

Creating custom:
from contextlib import contextmanager

@contextmanager
def get_db_session():
    session = SessionLocal()
    try:
        yield session
    finally:
        session.close()

with get_db_session() as db:
    users = db.query(User).all()
# Session automatically closed

Used extensively in database connection management."

**Q9: What is list comprehension?**

"Concise way to create lists.

Traditional:
squares = []
for i in range(10):
    squares.append(i**2)

List comprehension:
squares = [i**2 for i in range(10)]

With condition:
even_squares = [i**2 for i in range(10) if i % 2 == 0]

## POSTGRESQL QUESTIONS

**Q20: What are indexes and when should you use them?**

"Indexes speed up searches but slow down writes.

Without index: Full table scan (slow for large tables)
With index: Direct lookup (fast)

When to index:
- Frequently searched columns (WHERE clause)
- High cardinality (many unique values)
- Join columns (foreign keys)

NOT when:
- Low cardinality (e.g., boolean columns)
- Frequently updated columns
- Rarely searched columns

In RentalHunt, strategic indexing made search 30x faster."

---

**Q21: What is N+1 query problem?**

"Fetching one row, then fetching related rows one-by-one.

N+1 Problem:
properties = db.query(Property).all()  # 1 query
for property in properties:
    landlord = db.query(User).get(property.landlord_id)  # 1000 queries!

Solution: Eager loading
from sqlalchemy.orm import joinedload
properties = db.query(Property).options(
    joinedload(Property.landlord)
).all()  # Only 2 queries!

Result: 10+ seconds to 200ms (50x improvement)"

---

**Q22: What is a transaction?**

"A transaction groups multiple operations that must all succeed or all fail.

Example (bank transfer):
with db.begin():
    account_a.balance -= 1000  # Debit A
    account_b.balance += 1000  # Credit B
    # If B fails, A is also rolled back

ACID Properties:
- Atomicity: All or nothing
- Consistency: Data remains valid
- Isolation: Transactions don't interfere
- Durability: Committed data persists"

---

**Q23: What is database partitioning?**

"Splitting large tables into smaller pieces for performance.

Horizontal partitioning (by rows):
- Properties table partitioned by region
- Canada data in partition 1
- UAE data in partition 2
- Europe data in partition 3

Benefit: Queries only scan relevant partition

Vertical partitioning (by columns):
- Separate frequently accessed columns from rarely accessed
- Property basics in one table
- Property details (large text) in another"

---

**Q24: What is a database index type (B-tree, Hash, GIN)?**

"B-tree (default): Ordered, good for range queries
Example: WHERE price BETWEEN 1000 AND 2000

Hash: Exact match only, faster than B-tree for equality
Example: WHERE id = 123

GIN (Generalized Inverted Index): Full-text search, arrays
Example: WHERE search_vector @@ to_tsquery('python')

HNSW (for vectors): Approximate nearest neighbor search
Example: Vector similarity search in AI applications

In World Bank project, we use HNSW for embedding similarity search."

---

**Q25: What is connection pooling?**

"Reusing database connections instead of creating new ones.

Without pooling:
- Each request opens connection (expensive)
- 100 requests = 100 connections
- Connection overhead: 10-50ms each

With pooling (SQLAlchemy):
engine = create_engine(
    DATABASE_URL,
    pool_size=10,        # Max connections
    max_overflow=5,      # Extra connections if needed
    pool_timeout=30      # Wait time if pool full
)

Benefit: Connections are reused, much faster"

**Q31: What's the difference between Redis SET and SETEX?**

"SET: Store value without expiry
SETEX: Store value with expiry (seconds)

redis.set('user:123', data)         # Never expires
redis.setex('user:123', 3600, data) # Expires in 1 hour

For caching: Always use SETEX to prevent stale data"

---

**Q32: How do you handle cache invalidation?**

"Three strategies:

1. TTL (Time-to-Live): Expire automatically after time
   redis.setex('property:123', 3600, data)

2. Event-based: Delete/update when data changes
   redis.delete('property:123')  # When property updated

3. Cache-Aside (lazy loading): Update on cache miss
   cached = redis.get(key)
   if not cached:
       data = db.query(...)
       redis.setex(key, 3600, data)
   return cached or data

Challenge: Cache stampede (many requests hitting DB when cache expires)
Solution: Stale-while-revalidate pattern"

## AUTHENTICATION QUESTIONS

**Q50: How does JWT work?**

"JWT (JSON Web Token) is a stateless authentication mechanism.

Structure: header.payload.signature

Payload example:
{
  'sub': 'user_123',
  'email': 'user@example.com',
  'role': 'LANDLORD',
  'region': 'ca',
  'exp': 1625101200
}

How it works:
1. User logs in with credentials
2. Server creates JWT with user info
3. Returns JWT to client
4. Client sends JWT in every request
5. Server validates JWT signature (no DB lookup needed!)

Advantages:
- Stateless (no server-side session storage)
- Scalable (any server can validate)
- Contains user info (no DB lookup for basic info)"

---

**Q51: What's the difference between authentication and authorization?**

"Authentication: Who are you? (Login, verify identity)
Authorization: What can you do? (Permissions, roles)

Example:

# Authentication
def authenticate(token: str):
    payload = jwt.decode(token, SECRET_KEY)
    return payload['user_id']  # Proves who you are

# Authorization
def authorize(user_id: int, resource: str):
    user = db.query(User).get(user_id)
    if user.role == 'ADMIN':
        return True  # Can do anything
    if resource == 'own_profile':
        return True  # Can access own data
    return False  # Cannot access other's data"

---

6. Use access token to get user info

In RentalHunt, we use Firebase Auth which handles OAuth2 for us."

## LANGCHAIN/AI QUESTIONS

**Q60: What is RAG (Retrieval Augmented Generation)?**

"RAG combines retrieval (finding relevant documents) with generation (LLM response).

Without RAG:
- LLM only knows what it was trained on
- Can hallucinate facts
- Doesn't know your private documents

With RAG:
1. Store your documents in vector database
2. When user asks question, find relevant documents
3. Give relevant documents to LLM as context
4. LLM generates answer based on your documents

In World Bank project:
- Store policy documents in FAISS
- User asks question → retrieve relevant documents
- LLM analyzes those specific documents
- Response is grounded in actual documents"

---

**Q61: What is a vector database?**

"Vector database stores embeddings (numerical representations of text) and enables similarity search.

How it works:
1. Convert text to vector (list of numbers)
   'What is poverty?' → [0.23, -0.45, 0.78, ...]

2. Store in vector database with original text

3. When searching:
   - Convert query to vector
   - Find similar vectors (cosine similarity)
   - Return matching texts

In World Bank project:
- Documents converted to embeddings
- User query converted to embedding
- Find semantically similar document chunks
- Pass to LLM for analysis"

---

**Q62: What is LangGraph?**

"LangGraph is framework for building stateful, multi-step AI workflows.

Key Features:
- State management (agent remembers previous steps)
- Checkpointing (resume interrupted workflows)
- Multi-agent coordination
- Human-in-the-Loop support

Example workflow:
1. User submits query
2. Router agent decides what to do
3. Research agent retrieves documents
4. Validation agent cross-checks facts
5. HITL checkpoint: human reviews
6. Summarization agent creates response

In World Bank project: Reduced 2-3 day analyst work to 10-15 minutes."

---

**Q63: What is the ReAct pattern in AI agents?**

"ReAct = Reasoning + Acting

Agent thinks about what to do, then does it, then thinks again.

Cycle:
Thought: 'I need to find information about poverty rates'
Action: search_documents('poverty rates')
Observation: [retrieved documents]
Thought: 'I found the info, now I need to validate it'
Action: validate_facts([facts])
Observation: [validation results]
Thought: 'Validated. Now I can summarize'
Action: summarize([validated_facts])

Why it's powerful:
- Agent can reason step by step
- Can use tools based on what it finds
- Can self-correct"

---

**Q64: How do you prevent LLM hallucinations?**

"Hallucinations = AI stating facts that aren't true.

Prevention strategies:

1. Retrieval-Augmented Generation (RAG)
   - Every response grounded in retrieved documents
   - LLM can only use information from documents

2. Citation requirement
   - Force LLM to cite sources for every claim
   - Can validate if source actually says that

3. Validation agent
   - Second agent cross-checks facts
   - Identifies contradictions or unsupported claims

4. Confidence thresholds
   - If confidence < 70%, flag for human review
   - Don't auto-approve uncertain claims

5. Structured output
   - Require specific format (JSON with sources)
   - Easier to validate and verify

In World Bank project: Reduced hallucination rate from 8% to <0.5%"

**Q70: What is Celery?**

"Celery is a distributed task queue for Python.

When to use:
- Long-running tasks (don't block HTTP response)
- Scheduled tasks (cron jobs)

- Email sending, report generation

**Q71: What's the difference between Celery and FastAPI BackgroundTasks?**

"FastAPI BackgroundTasks:
- Runs in same process
- Good for quick tasks
- Not distributed
- No retry on failure

Celery:
- Separate worker processes
- Good for heavy tasks
- Distributed (multiple workers)
- Automatic retry

Rule of thumb:

## SYSTEM DESIGN QUESTIONS

**Q90: How would you design a URL shortener (like bit.ly)?**

"Requirements:
- Short links (e.g., short.ly/abc123)
- Redirect to original URL
- Track click count
- Handle millions of URLs

Design:

1. API:
   POST /shorten {'url': 'long-url.com'} → {'short': 'abc123'}

2. Database:
   urls table:
   - id (PK)
   - short_code (6 chars, unique)
   - original_url
   - created_at
   - click_count

3. Short code generation:
   - Option 1: Random 6-char alphanumeric
   - Option 2: MD5 hash of URL, take first 6 chars

4. Scaling:
   - Read-heavy: Cache popular URLs in Redis
   - Write-heavy: Database write replicas
   - Analytics: Separate table/database for click events"

---

**Q91: How would you design a real-time chat system?**

"Requirements:
- Messages sent/received in real-time
- 10K concurrent users
- Message history

Design:

1. WebSocket for real-time:
   - User connects: ws://api.com/ws/{user_id}
   - Server pushes messages immediately

2. Database:
   messages table:
   - id, sender_id, receiver_id, content, timestamp, read_at

3. Scaling WebSockets:
   - Problem: Server A has user 1, Server B has user 2
   - Server A doesn't know user 2's connection
   - Solution: Redis Pub/Sub
     - User 1 sends message → Server A publishes to Redis
     - Server B subscribed to user 2's channel → Delivers message

4. Message history:
   - GET /conversations/{user_id}?page=1 → Paginated history
   - Index on (sender_id, receiver_id, timestamp)"

**Q92: How would you design a rate limiter?**

"Rate limiting: Prevent API abuse.

Requirements:
- Max 100 requests per user per minute
- Block temporarily if exceeded

Implementation using Redis:

def rate_limit(user_id: str, limit: int = 100):
    key = f'rate_limit:{user_id}:{current_minute()}'
    count = redis.incr(key)

    if count == 1:
        redis.expire(key, 60)  # Expire after 1 minute

    if count > limit:
        raise HTTPException(429, 'Rate limit exceeded')

@app.middleware('http')
async def rate_limit_middleware(request, call_next):
    user_id = get_user_id(request)
    rate_limit(user_id)
    return await call_next(request)

Algorithms:
- Fixed window: Simple, but burst at window edge
- Sliding window: Smoother, more accurate
- Token bucket: Allows bursting but limits average"
